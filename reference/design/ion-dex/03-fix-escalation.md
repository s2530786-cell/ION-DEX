# 修复层级图 (Fix Escalation)

## 层级定义

### L0: 自修复 (Self-Heal)
- **触发**: 编译错误、类型错误、lint 警告
- **责任人**: 发现问题的角色
- **时限**: 5 分钟内
- **动作**: 直接修复 → tsc 验证 → commit
- **升级条件**: 5 分钟未修复 → L1

### L1: 角色内修复 (Role-Internal)
- **触发**: L0 升级 / 角色职责范围内的 bug
- **责任人**: 当前角色
- **时限**: 15 分钟内
- **动作**: 分析根因 → 修复 → 自测 → commit
- **升级条件**: 15 分钟未修复 / 涉及其他角色领域 → L2

### L2: 跨角色协作 (Cross-Role)
- **触发**: L1 升级 / 跨角色 bug / 设计冲突
- **责任人**: 相关角色 + Code Review Engineer (CRE)
- **时限**: 30 分钟内
- **动作**: 相关角色会诊 → 确定修复方案 → 分派执行 → CRE review → commit
- **升级条件**: 30 分钟未达成共识 / 涉及 Design Tokens 或 Grid Harness → L3

### L3: 架构级修复 (Architecture)
- **触发**: L2 升级 / Design Tokens 变更 / Grid Harness 变更 / .cursorrules 变更
- **责任人**: DTE + GSE + CG + CRE 联合
- **时限**: 60 分钟内
- **动作**: 架构评审 → 方案设计 → 实施 → 全量回归测试 → 双 R3 签批
- **升级条件**: 60 分钟未解决 / 影响 P0 阀门 → L4

### L4: 紧急制动 (Emergency Brake)
- **触发**: L3 升级 / 生产环境异常 / 安全漏洞 / 构建失败阻塞
- **责任人**: 全团队 (18 角色)
- **时限**: 立即响应，120 分钟内解决
- **动作**: 
  1. 立即停止所有 P0-P2 任务
  2. 全团队会诊
  3. 热修复分支
  4. 全量测试
  5. 双 R3 签批
  6. 部署 + 监控
- **升级条件**: 120 分钟未解决 → 通知 Master

## SLA 总表

| 层级 | 响应 | 解决 | 升级触发 | 审批 |
|------|------|------|----------|------|
| L0 | 即时 | 5min | 5min | 无 |
| L1 | 即时 | 15min | 15min | 自签 |
| L2 | 5min | 30min | 30min | CRE |
| L3 | 10min | 60min | 60min | DTE+GSE+CG |
| L4 | 即时 | 120min | 120min | 全团队+Master |

## 升级链路图
```
L0 (自修复)
  ↓ 5min未解决
L1 (角色内)
  ↓ 15min未解决 / 跨角色
L2 (跨角色协作)
  ↓ 30min未解决 / 涉及核心架构
L3 (架构级)
  ↓ 60min未解决 / 紧急
L4 (紧急制动)
  ↓ 120min未解决
Master 介入
```

## 缰绳规则
- 每次升级必须记录：触发原因、升级时间、当前状态
- L3+ 修复必须写 post-mortem
- 同一类型 bug 触发 L3 超过 3 次 → 自动升级为架构改进任务
- 任何角色可发起升级，但必须附带 L-1 层级的尝试记录
