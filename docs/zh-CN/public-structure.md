# 公开结构

> 面向中文读者的 ION DEX 公开组织、治理与披露边界说明。

## 项目现状

**ION DEX 当前由 Master 独立开发。**

这不是委员会工程，不是换皮 fork，也不是短期代币叙事工具，而是一套长期主义、方向连续的系统工程。

## 开发哲学

### 公开开发

- 源代码公开在 GitHub
- 文档公开在 `docs/`
- 路线图公开在 README 与路线图指南中
- 合约地址在正式部署后发布并可在浏览器验证

### 私有战略层

以下内容可以保留在私有层：

- 未发布模块的精确 rollout 时间
- 竞争策略
- 内部仲裁与风控细节
- 尚未公开的合作信息

## 治理模型

### 当前阶段：单操作者

当前可视作 **P0–P1** 期间：

- Master 负责战略决策
- 开发方向沿 P0–P10 推进
- 涉及真实资金前，高风险合约升级应配 timelock / multisig

### 未来阶段：社区治理

在更成熟阶段，可能逐步引入：

- 提案系统
- 投票机制
- 多签国库
- 治理 timelock

关键原则是：**治理扩大不能以安全性和系统一致性为代价。**

## 审计与安全结构

真实资金上线前，应至少满足：

1. 内部审查
2. 外部专业审计
3. 压力 / 攻击模拟
4. 公开审计报告

## 收益分配边界

公开收益分配材料目前都应理解为**草案经济设计**，而不是已审计、已部署、已冻结的主网分配规则。

## 透明度原则

### 哪些内容应该可验证

- 已部署模块的交易历史
- 已生效的销毁事件
- 已上线的质押仓位
- 已发布的合约部署记录

### 哪些内容会公开

- README
- 路线图
- 白皮书
- API 文档
- 已正式发布的合约地址

### 哪些内容不一定公开

- 精确未发布排期
- 竞争打法
- 内部仲裁流程细节
- 未宣布合作细节

## 与公共秩序兼容

目标是做到：

- 持续身份提供问责，但不过度滑向监控
- 不可变历史提供信任，但不依赖中心控制
- 证明系统提供可验证性，而不以许可为前提
